## 第 1 章 · 大型网站架构演化

> 核心问题：**网站从小到大的过程中，架构如何一步步演化？**

### 演化路径（由简到繁）

| 阶段 | 架构特征 | 解决的核心问题 |
|------|----------|----------------|
| 1. 单机部署 | 应用 + 文件 + 数据库在同一台服务器 | 从零到一，快速上线 |
| 2. 应用与数据分离 | 应用服务器 / 文件服务器 / 数据库服务器 三台独立 | 单一服务器性能瓶颈 |
| 3. 使用缓存 | 引入本地缓存（Ehcache）→ 分布式缓存（Redis/Memcached） | 数据库读压力过大（二八定律：80% 的访问落在 20% 的数据上） |
| 4. 应用服务器集群 | 多台应用服务器 + 负载均衡（Nginx/LVS） | 单台应用服务器并发能力有限 |
| 5. 数据库读写分离 | 主库写 + 从库读，主从复制 | 数据库读写相互竞争 |
| 6. 反向代理 + CDN | CDN 缓存静态资源到边缘节点，反向代理加速 | 用户访问速度慢（跨地域），带宽压力 |
| 7. 分布式文件 + 分布式数据库 | 文件系统（HDFS/MinIO）、数据库（水平分库分表） | 单表数据量巨大，单机存储空间不足 |
| 8. NoSQL + 搜索引擎 | Redis/MongoDB/Elasticsearch 补充关系型数据库 | 复杂查询与全文搜索需求 |
| 9. 业务拆分 / 微服务 | 按业务领域拆成独立服务（订单/用户/商品等） | 单体应用臃肿，开发效率下降 |

### 演化的核心驱动力
- **用户规模增长** → 性能、可用性需求提升
- **业务复杂度提高** → 扩展性、伸缩性需求提升
- **数据量膨胀** → 存储架构必须变革

### 重要认知
> 没有最好的架构，只有最适合当前业务规模的架构。**架构是演化出来的，不是设计出来的**。

## 第 2 章 · 大型网站架构模式

> 核心问题：**有哪些可复用的架构模式？**

### 九大架构模式

| 模式 | 核心思想 | 举例 |
|------|----------|------|
| **分层** | 横向切分：表现层 → 应用层 → 服务层 → 数据层 | MVC、OSI 七层模型 |
| **分割** | 纵向切分：按业务功能拆分模块 | 订单模块、用户模块、商品模块 |
| **分布式** | 同一个功能的不同部分部署在不同服务器上 | 分布式缓存、分布式数据库、分布式文件系统 |
| **集群** | 同一个功能的多台服务器在一起工作 | 应用服务器集群、缓存集群 |
| **缓存** | 用空间换时间，存储计算结果或数据副本 | CDN → 反向代理 → 本地缓存 → 分布式缓存 的多级缓存体系 |
| **异步** | 非实时处理，削峰填谷 | 消息队列（MQ）处理下单后的发短信、写日志 |
| **冗余** | 多副本部署，避免单点故障 | 数据库主从、冷备/热备、异地多活 |
| **自动化** | 减少人工干预，提高效率 | 自动构建、自动部署、自动扩容、自动故障转移 |
| **安全** | 保护数据和系统免受攻击 | XSS/CSRF 防护、SQL 注入防护、密码加盐哈希 |

### 模式之间的关系
- **分层 + 分割** = 横纵切分，构成模块矩阵
- **分布式 + 集群** = 解决单点性能瓶颈
- **缓存 + 异步** = 提升性能和用户体验的两大利器
- **冗余 + 自动化** = 可用性的基础保障

## 第 3 章 · 大型网站核心架构要素

> 核心问题：**评价一个架构好坏的维度有哪些？**

### PASES 五要素

| 要素 | 英文 | 核心问题 | 关键手段 |
|------|------|----------|----------|
| **性能** | Performance | 用户等得快吗？ | 缓存、异步、CDN、多级优化 |
| **可用性** | Availability | 系统多久挂一次？ | 冗余、故障转移、监控告警 |
| **伸缩性** | Scalability | 加机器能不能扛更大的量？ | 无状态化、水平拆分、一致性哈希 |
| **扩展性** | Extensibility | 加新功能要不要动老代码？ | 模块化、消息队列、服务框架、开放接口 |
| **安全性** | Security | 数据不会被盗/被篡改吗？ | 加密、身份认证、权限控制、审计 |

### 要素之间的权衡
- **不可能五个同时做到极致**——资源有限，需根据业务阶段取舍
- 常见矛盾：
  - 追求安全性 → 引入加密/认证 → 性能下降
  - 追求伸缩性 → 分布式改造 → 复杂度增加，可用性挑战增大
  - 追求高性能 → 大量缓存 → 数据一致性问题

## 第 4 章 · 高性能架构

> 核心问题：**从用户浏览器到底层存储，每一层如何优化？**

### 三层优化全景

#### 1️⃣ 前端层（用户侧）
| 手段 | 原理 | 效果 |
|------|------|------|
| CDN 加速 | 静态资源缓存到离用户最近的 CDN 节点 | 减少网络延迟 |
| 动静分离 | 静态资源（CSS/JS/图片）独立部署 | 静态资源可独立 CDN 加速 |
| 浏览器缓存 | 设置 Cache-Control / ETag 头 | 减少重复请求 |
| 页面优化 | 减少 HTTP 请求（合并 CSS/JS）、压缩、懒加载 | 减少传输量 |
| 反向代理 | Nginx/Squid 缓存部分动态页面输出 | 减轻应用服务器压力 |

#### 2️⃣ 应用层（服务端）
| 手段 | 原理 | 关键点 |
|------|------|--------|
| **本地缓存** | 进程内缓存，访问最快 | 受 JVM 内存限制，多机数据不一致 |
| **分布式缓存** | Redis/Memcached，集中式缓存 | 数据一致，但有网络开销 |
| **异步处理** | MQ 解耦，非关键路径异步化 | 削峰填谷，提高吞吐量 |
| **集群部署** | 多台应用服务器水平扩展 | 要求应用无状态 |
| **代码优化** | 合理的数据结构、连接池、线程池 | 从根上提效 |

#### 3️⃣ 存储层（数据）
| 手段 | 原理 |
|------|------|
| 读写分离 | 主库写 + 多从库读，分摊读压力 |
| 分库分表 | 水平拆分（按 user_id 取模） + 垂直拆分（按业务字段） |
| NoSQL | Redis/MongoDB 等，放弃部分 ACID 换取高性能 |
| 搜索引擎 | Elasticsearch 做全文检索，不用 `LIKE` 扫全表 |

### 核心原则
> **性能优化的关键是"离用户越近越好，数据越少越准越好"**。

## 第 5 章 · 高可用架构

> 核心问题：**怎么保证系统 7×24 小时不挂？**

### 高可用的核心公式

```
可用性 = 冗余（Redundancy） + 故障转移（Failover）
```

### 分层可用性设计

#### 应用层（最简单的冗余）
- **无状态化**：Session 不放在应用服务器里，放到 Redis/共享存储
- **负载均衡**：Nginx/HAProxy/云厂商 SLB，心跳检测自动踢掉故障节点
- 核心原则：**应用服务器可以随时被替换**

#### 服务层
- **服务发现**：注册中心（Nacos/Zookeeper/Eureka），服务上下线自动感知
- **熔断降级**：故障服务自动熔断（Sentinel/Hystrix），防止级联雪崩
- **限流**：对超出处理能力的请求直接拒绝，保护系统不被冲垮

#### 存储层（最难做到高可用的层）
| 架构 | 模式 | 优缺点 |
|------|------|--------|
| 主备（Master-Backup） | 一台工作，一台待命 | 简单但浪费资源，切换有延迟 |
| 主从（Master-Slave） | 主写从读，从库可分担查询 | 主挂了从需手动提升 |
| 主主（Master-Master） | 双主对等，互相同步 | 写冲突风险大，慎用 |
| 多副本集群 | Paxos/Raft 共识算法选主 | 强一致但性能有损耗 |

#### CAP 理论 vs BASE 理论
| 理论 | 含义 | 适用场景 |
|------|------|----------|
| **CAP** | C（一致性）A（可用性）P（分区容错）三者最多同时满足两个 | 分布式系统的硬约束 |
| **BASE** | Basically Available（基本可用）+ Soft state（软状态）+ Eventually consistent（最终一致） | 大型互联网的务实选择 |

> **大型互联网系统普遍选择 AP（牺牲强一致，保障可用性和分区容错），用最终一致性满足业务需求。**

## 第 6 章 · 伸缩性架构

> 核心问题：**用户量翻了 10 倍，架构能不能通过加机器扛住？**

### 垂直伸缩 vs 水平伸缩

| 方式 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **垂直伸缩**（Scale Up） | 服务器升级 CPU/内存/磁盘 | 简单，不需要改架构 | 有物理上限，成本指数增长 |
| **水平伸缩**（Scale Out） | 加更多服务器 | 理论无限扩展，成本线性 | 需要架构支持，复杂度高 |

> **大型网站必须走水平伸缩路线。**

### 各层的伸缩性方案

| 层次 | 伸缩方案 | 关键技术点 |
|------|----------|------------|
| **应用层** | 应用服务器集群 | **无状态化是前提**——任何请求可以被任何节点处理 |
| **缓存层** | Redis 集群 | **一致性哈希**：加节点时尽量少迁移数据，减少缓存雪崩 |
| **数据库** | 分库分表 | 核心难题：数据路由规则（如按 user_id 取模）、跨分片查询、数据迁移 |
| **文件存储** | 分布式文件系统 | HDFS/MinIO，文件自动分布到多台存储节点 |

### 数据库伸缩的难点
- **分片规则**：决定了数据分布均匀性和查询便利性的平衡
- **跨分片 Join**：关系型数据库最大的软肋 → 方案：冗余字段 / 应用层 Join / NoSQL
- **数据迁移**：扩缩容时如何平滑迁移数据（双写 + 历史数据逐步迁移）

### 核心原则
> **伸缩性做得好不好，关键看"加机器"这个操作是否需要改代码、停服务。**

## 第 7 章 · 可扩展架构

> 核心问题：**加一个新功能，能不能不动老代码？**

### 可扩展 ≠ 伸缩性
- **伸缩性** = 加机器就能扛住更大的量（Scale Out）
- **可扩展性** = 加功能不用大改老系统（Add Features with Minimal Impact）

### 实现可扩展的核心手段

#### 1. 模块化 / 组件化
- 用接口定义契约，用实现类提供能力
- **依赖倒置**：高层模块不依赖低层模块，两者都依赖抽象（接口）
- 开闭原则（OCP）：对扩展开放，对修改关闭

#### 2. 分布式消息队列（解耦利器）
```
业务系统 A → MQ → 业务系统 B/C/D
           ↓ 异步 ↓
         A 不管 B/C/D 是否在线
```
- 新增一个消息消费方，完全不碰消息生产方的代码
- 常见选型：Kafka（大数据量） / RocketMQ（事务消息） / RabbitMQ（路由灵活）

#### 3. 分布式服务框架（服务化）
| 框架 | 核心能力 |
|------|----------|
| **Dubbo / Spring Cloud** | RPC 调用、服务注册发现、负载均衡 |
| **服务总线（ESB）** | 集中式路由、协议转换（重型，用得少） |
| **Service Mesh** | Sidecar 代理模式（Istio/Envoy），业务零侵入 |

#### 4. 开放平台
- 对外暴露 Open API，让第三方可以基于平台开发新功能
- 本质：把"内部扩展"的能力开放给外部

### 可扩展架构的本质
> **用"模块间松耦合"换"功能的随意组合"。模块内部高内聚，模块之间靠接口/消息通信。**

### 可扩展 vs 微服务
- 微服务是可扩展架构的一种**极致实现**：每个服务都是独立部署的最小单元
- 但微服务不是免费的午餐——引入分布式事务、网络延迟、运维复杂性等代价

## 第 8 章 · 安全架构

> 核心问题：**网站如何抵御攻击，保护数据不被窃取或篡改？**

### 常见 Web 攻击与防御

| 攻击类型 | 原理 | 防御手段 |
|----------|------|----------|
| **XSS（跨站脚本）** | 恶意脚本注入页面，盗取 Cookie/会话 | 输出转义（HtmlEncode）、CSP 头、HttpOnly Cookie |
| **SQL 注入** | 拼接恶意 SQL 参数，越权操作数据库 | 参数化查询（PreparedStatement）、ORM 框架、输入校验 |
| **CSRF（跨站请求伪造）** | 诱导用户点击，借用其身份发请求 | Token 验证、Referer 校验、SameSite Cookie |
| **文件上传漏洞** | 上传可执行脚本到服务器 | 文件类型校验、重命名、存储隔离 |
| **DDoS** | 海量请求耗尽带宽/连接数 | 流量清洗、CDN 分散、限流 |

### 信息加密三件套

| 加密类型 | 特点 | 典型应用 |
|----------|------|----------|
| **单向散列** | 不可逆，固定长度输出 | MD5/SHA-256 存密码（加盐！） |
| **对称加密** | 同一密钥加解密，速度快 | AES/DES，会话加密、存储加密 |
| **非对称加密** | 公钥加密 + 私钥解密，安全但慢 | RSA/ECC，数字签名、HTTPS 握手 |

> **密钥管理比算法更重要**——密钥硬编码在代码里 = 门锁钥匙插在锁孔上。

### 信息过滤与风险控制
- **敏感词过滤**：文本匹配（Trie 树）→ 分类算法（贝叶斯）→ 黑名单
- **风控**：规则引擎 + 机器学习识别异常交易（异地登录、短时间大量下单等）

## 第 9 章 · 淘宝网的架构演化案例

> **从 LAMP 单体应用到分布式服务化架构的实战历程**

### 演化关键节点

| 阶段 | 架构特点 | 关键举措 |
|------|----------|----------|
| **早期（PHP）** | LAMP 单体，MySQL + 文件存储 | 快速上线验证商业模式 |
| **Java 改造** | 从 PHP 迁移到 Java | 工程化：IDE 支持、代码规范、自动化测试 |
| **服务化 1.0** | 按业务拆服务，通过 Web Service 调用 | 每个服务一个团队，独立开发部署 |
| **中间件建设** | 自研 RPC（HSF）、消息中间件（Notify）、分库分表（TDDL） | 服务框架标准化，降低分布式复杂度 |
| **去 IOE** | 替换 IBM 小型机 + Oracle + EMC 存储 | 用 PC 服务器 + MySQL + SSD 降低成本 |
| **云化（阿里云）** | 输出技术服务能力，多机房多活 | 服务外部客户的同时提升自身弹性 |

### 核心启示
- **架构随业务演进而非一步到位**——淘宝也不是一开始就是现在的架构
- **基础设施自研是规模化后的必然选择**——开源方案满足不了极致需求时再自研
- **服务化成功的关键**：组织架构同步调整（康威定律：系统架构反映团队沟通结构）

## 第 10 章 · 维基百科的高性能架构案例

> **全球 Top 10 网站，如何用最小的硬件成本服务海量用户？**

### 架构亮点

| 层面 | 方案 | 细节 |
|------|------|------|
| **Web 层** | 缓存优先 | Squid 反向代理 + APC（PHP 缓存）覆盖大部分请求 |
| **数据层** | MySQL + Memcached | 数据放 MySQL，缓存热点数据到 Memcached |
| **前端** | CDN（GeoDNS） | 根据用户地理位置分配最近的 CDN 节点 |
| **语言** | PHP（性能不够？） | 用 HipHop 编译器将 PHP 编译为 C++ 再运行 |
| **基础设施** | 全部使用开源软件 | LAMP + Squid + Linux-HA，零商业软件成本 |

### 核心启示
- **缓存是最便宜的扩容方式**——Wikipedia 的缓存命中率极高，数据库压力极小
- **简单方案 + 极致优化 > 复杂方案**——不追新技术，把现有工具用到极致
- **开源生态支撑全球级网站完全可行**

## 第 11 章 · Doris 海量分布式存储的高可用设计

> 核心问题：**一个服务于海量电商图片的存储系统如何做到永不丢数据？**

### Doris 是什么
- 淘宝自研的分布式 Key-Value 存储系统，用于存储商品图片、店铺装修等海量小文件
- 设计目标：**高可用 > 高性能 > 低成本**

### 高可用设计核心

| 策略 | 实现 |
|------|------|
| **多副本** | 每份数据保存 3 副本，分布在不同物理节点 |
| **副本一致性** | 写时同时写多副本，读时选择可用副本 |
| **故障自愈** | 节点故障 → 自动检测 → 副本不足时自动补充到 3 副本 |
| **版本控制** | 数据写入带版本号，解决并发写冲突 |
| **分级存储** | 热点数据 SSD → 温数据 SAS → 冷数据 SATA |
| **多活部署** | 不同 IDC 各部署一套，就近访问 |

### 架构分层
```
接入层（Router）→ 定位层（NameServer 路由表）→ 存储层（DataServer 实际存储）
```

> **高可用的基石不是 "不故障"，而是 "故障后自动恢复"。**

## 第 12 章 · 网购秒杀系统架构设计

> 核心问题：**10 万人同时抢 100 件 1 元商品，怎么不崩？**

### 秒杀的难点
- **瞬时并发极高**：秒杀开始的瞬间，流量是平时的几十到几百倍
- **超卖风险**：库存 100 件，卖出了 101 件就是灾难
- **公平性要求高**：先到先得，不能随机丢弃

### 秒杀架构全景

| 阶段 | 手段 | 作用 |
|------|------|------|
| **前端层** | 静态化页面 + 按钮防重复点击（置灰 + CDN 缓存静态资源） | 减少 99% 的动态请求 |
| **网关层** | 秒杀时间点前拦截请求，返回"等待中" | 挡住无效请求 |
| **服务层** | 请求排队（MQ）+ 限流 + 熔断 | 削峰，保护后端 |
| **业务层** | Redis 缓存库存 + Lua 原子扣减 | 在内存中判断库存，比数据库快 1000 倍 |
| **数据层** | 最终异步写库（MQ → 批量更新 MySQL） | 数据库只负责持久化，不参与抢购逻辑 |

### 防超卖的核心

```
关键：减库存操作必须是原子性的
方案：Redis DECR（原子递减）+ 返回值判断（>=0 表示抢到）
     或者 MySQL UPDATE ... WHERE stock >= 1（行锁保证原子性）
```

### 秒杀分层架构

```
静态页面（CDN）
    ↓ 按钮 JS 控制 + 秒杀地址动态生成（防外挂）
Nginx 限流（limit_req_zone → 令牌桶算法）
    ↓
Redis 库存预扣（Lua 脚本保证原子性）
    ↓
MQ 异步写订单 → 支付倒计时（15 分钟内未支付则释放库存）
```

> **秒杀的本质是一场"流量漏斗"：99.9% 的请求在前几层被无害拦截，最终只有 0.1% 的请求真正竞争库存。**

## 第 13 章 · 大型网站典型故障案例

> **所有大故障背后都有简单到可笑的原因——但事后回顾才觉得简单。**

### 九类典型故障

| 故障类型 | 真实案例 | 教训 |
|----------|----------|------|
| **日志文件满** | 日志写满磁盘 → 所有写操作失败 → 全站不可用 | 日志轮转 + 磁盘监控告警 |
| **数据库高并发死锁** | 热点商品行锁竞争 → 死锁 → DB 连接池耗尽 | 热点数据放缓存，减少 DB 直接竞争 |
| **分布式锁故障** | 锁超时释放 → 另一个线程获取锁 → 并发操作同一数据 | 锁续期（watchdog）、不可重入锁注意逻辑 |
| **缓存雪崩** | 大量缓存同时过期 → 请求全部打到 DB → DB 宕机 | 过期时间加随机值，热点数据永不过期 |
| **全量拉取** | 定时任务全量拉取 DB → 慢查询拖垮从库 → 影响所有读请求 | 分批拉取 + 游标，错开高峰期 |
| **网络超时设置不当** | 服务 A 调 B 超时 10s → 线程池满 → A 不可用 → 级联故障 | 超时设短（1-3s），快速失败 |
| **单点故障** | 单一负载均衡器故障 → 流量全断 | 所有环节必须有冗余（包括你的冗余方案本身） |
| **发布回滚失败** | 新版本有 Bug，回滚脚本不可用 | 回滚是发布流程的一部分，必须测试 |
| **配置错误** | 修改了数据库连接配置 → 所有服务起不来 | 配置中心 + 灰度发布 + 配置变更走审批 |

### 核心启示
> **故障不可怕，可怕的是同一个原因导致两次故障。每次故障必须深入根因 + 举一反三。**

## 第 14 章 · 架构师领导艺术

> 架构师不是只管技术的独行侠，而是技术方向的引领者。

### 领导三要素

| 要素 | 怎么做 | 反面教材 |
|------|--------|----------|
| **关注人** | 了解每个成员的长处和成长诉求，合理分配任务 | 只管技术不管人，团队离心 |
| **共享蓝图** | 反复讲技术愿景，让团队看到一致的方向 | 只有自己知道目标，团队被动执行 |
| **成就他人** | 把功劳给做事的同学，自己做背后推手 | 独占功劳，团队没有积极性 |

### 关键原则
- **学会妥协**：坚持技术方向，但在实现路径上灵活
- **共同参与**：让团队参与架构决策，而非一言堂——参与感带来责任感
- **发掘人的优秀**：发现每个人的闪光点并放大

> **架构师的成就 = 团队的成功，而不是个人写了多少代码。**

## 第 15 章 · 网站架构师职场攻略

> 有技术不够，还要能让技术落地。

### 三步走策略

```
发现问题 → 提出问题 → 解决问题
```

| 步骤 | 关键动作 | 误区 |
|------|----------|------|
| **发现问题** | 深入一线看代码、看监控、看故障报告，找到真正的痛点 | 坐在办公室拍脑袋想痛点 |
| **提出问题** | 用数据说话（宕机时长、性能指标对比），量化改进收益，争取支持 | 只说"太慢了"、"太烂了"，没有量化证据 |
| **解决问题** | 分阶段落地，每个阶段有可验证的成果，积累信任 | 一步到位大改造，中间没交付，失去支持 |

### 关键认知
- **技术与业务的平衡**：技术方案要能翻译成业务价值（省了多少钱/提升了多少转化率）
- **向上管理**：让老板看到技术的价值，争取时间与资源
- **时机很重要**：在大故障发生后推动高可用改造，比平时更有说服力

> **好的架构师不是技术最强的人，而是把合适的技术在合适的时间推上去的人。**

## 第 16 章 · 漫话网站架构师

> 回顾全书，从不同维度勾勒架构师的画像。

### 架构师的七种分类维度

| 维度 | 类型 | 特征 |
|------|------|------|
| **按作用分** | 设计型 / 救火型 / 公关型 | 不是每个架构师都做设计，也有专门处理线上事故的 |
| **按效果分** | 建设型 / 破坏型 | 好的架构带来增长，坏的架构拖累业务 |
| **按职责角色分** | 系统架构师 / 应用架构师 / 数据架构师 / 运维架构师 | 不同层级的关注点不同 |
| **按关注层次分** | 技术架构 / 业务架构 / 组织架构 | 层次越高，越关注"人"和"流程" |
| **按口碑分** | 有口碑的 / 无口碑的 | 代码和设计是会说话的 |
| **按非主流方式分** | 学院派 / 江湖派 | 有的靠理论驱动，有的靠实战经验驱动 |

### 架构师的修养
- **持续学习**：技术更新太快，不学习就淘汰
- **独立思考**：不盲从方案、不追热门，选择最适合业务的
- **保持好奇**：对新技术保持关注，但不盲目引入
- **注重实践**：好的架构师用代码说话

> **架构师 = 技术深度 × 业务理解 × 沟通能力 × 持续学习**

## 全书总结：十六章的核心脉络

```
第1篇 概述（第1~3章）
    ├── 第1章 演化：从单机到分布式，架构是一步步"长"出来的
    ├── 第2章 模式：九大模式是解决共性问题的最佳实践
    └── 第3章 要素：性能·可用性·伸缩性·扩展性·安全性——五大维度评价架构

第2篇 架构（第4~8章）—— 五大要素逐一深入
    ├── 第4章 高性能：前端→应用→存储，三层优化，离用户越近越好
    ├── 第5章 高可用：冗余 + 故障转移，存储层最难高可用，CAP/BASE 做取舍
    ├── 第6章 伸缩性：水平扩展是王道，无状态化是前提，一致性哈希是关键
    ├── 第7章 可扩展：MQ 解耦、服务框架、开放接口，让加功能不改老代码
    └── 第8章 安全：XSS/CSRF/SQL注入、加密三件套、风控

第3篇 案例（第9~13章）—— 别人走过的路
    ├── 第9章 淘宝：从 PHP → Java → 服务化 → 去IOE → 云化
    ├── 第10章 维基百科：开源栈 + 极致缓存，用最少的硬件做最大的事
    ├── 第11章 Doris：三副本 + 自动故障恢复，数据永不丢失
    ├── 第12章 秒杀：流量漏斗，层层拦截，Redis 原子扣库存
    └── 第13章 故障：日志满、死锁、缓存雪崩……每个故障都是学费

第4篇 架构师（第14~16章）—— 做事的人
    ├── 第14章 领导艺术：关注人、共享蓝图、成就他人
    ├── 第15章 职场攻略：发现问题→提出问题→解决问题，用数据说话
    └── 第16章 人物画像：技术深度 × 业务理解 × 沟通能力 × 持续学习
```

> **一句话总结本书核心思想**：大型网站架构不是一步到位的设计，而是在应对业务量增长的过程中，不断通过**分层、缓存、异步、集群、冗余、自动化**等手段持续演化的结果。五个核心要素（性能、可用性、伸缩性、扩展性、安全性）是技术面的评价维度，而架构师的领导力、职场智慧和持续学习能力，则是将这些技术落地的人性面保障。
